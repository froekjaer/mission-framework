# Framework Stabilisation Criteria

**Status:** Normative release governance  
**Applies from:** Foundation v1.1

## Purpose

The stabilisation criteria prevent conceptual growth from outrunning evidence and implementation capacity.

They do not prohibit correction. They require that substantial expansion be justified.

## Stabilisation posture

During stabilisation:

- existing concepts may be clarified when ambiguity is evidenced
- defects and contradictions may be corrected
- validation tooling and conformance checks may be added
- reference missions may create local implementation patterns
- new Mission Core concepts require explicit evidence and admission review
- broad platform commitments remain provisional

## Entry criteria

The framework enters stabilisation when:

- a coherent foundation baseline exists
- independent review has been completed
- major review findings have been synthesised
- the dominant remaining risk is empirical rather than conceptual

Foundation v1.1 satisfies these entry criteria.

## Exit criteria

Stabilisation may conclude when all mandatory criteria below are met.

### 1. Reference validation

- at least one bounded Mission Solar Eclipse vertical slice is completed
- its proposition, baseline, measures and evidence are published
- the result is classified using the validation dispositions

### 2. Framework Findings

- material tensions discovered during the reference mission are recorded as Framework Findings
- each accepted finding has an accountable disposition
- rejected or deferred findings remain traceable

### 3. Reproducibility

- a cross-repository baseline manifest identifies exact versions
- an independent reviewer can locate the required artefacts
- the main mission reasoning can be reconstructed without private conversational context

### 4. Minimum conformance

Automated or repeatable checks verify at least:

- required canonical documents exist
- internal links and referenced repository locations resolve where technically possible
- version and status metadata are present for release-controlled documents
- reference missions do not silently redefine canonical terms
- generated publications can be traced to reviewed sources

### 5. Burden assessment

The programme records:

- participant effort attributable to framework use
- documentation or governance judged unnecessary
- skills required for independent use
- known dependence on the original author

### 6. Release decision

An accountable release decision states whether to:

- continue validation
- revise the framework
- admit new concepts
- begin broader platform evolution
- remain stabilised
- deprecate unsupported elements

## Evidence threshold for substantial expansion

A substantial semantic addition should normally require:

1. a documented problem in an actual mission or repeated implementation context
2. evidence that existing concepts cannot address the problem without distortion
3. comparison with a simpler clarification or implementation-local solution
4. a Framework Finding and Mission Core admission decision
5. identified validation consequences

## Exceptions

Urgent legal, safety, security or ethical corrections may proceed before full validation. The exception, rationale, owner and intended follow-up validation shall be recorded.

## Non-criteria

The following do not, by themselves, justify leaving stabilisation:

- number of documents produced
- number of concepts defined
- volume of code
- reviewer enthusiasm
- AI agreement
- platform feature count
- elapsed time

## Closing rule

Stabilisation ends because evidence demonstrates readiness, not because the programme becomes impatient with validation.
