# v0.2 — Semantic Foundation

> **Mission Framework is a semantic foundation for describing, governing and improving missions across domains.**

## Purpose of v0.2

Version 0.2 is the first semantic baseline of Mission Framework.

Its purpose is not to make the framework complete. Its purpose is to make the framework coherent enough that a person or machine can understand and apply its core semantics without access to the authors, private conversations, or undocumented assumptions.

The release is therefore defined by comprehension and interoperability rather than document count.

## Primary acceptance test

> **Can independent readers — human or machine — read v0.2 without conversation history and reach a materially compatible understanding of the framework's core concepts, relationships and rules?**

A successful v0.2 should allow independent readers to explain, with compatible meaning:

- what a Mission is;
- how Reality relates to models and evidence;
- how Observation, Evidence, Claim, Decision, Action, Outcome and Learning relate;
- how authority and accountability remain traceable when actions are delegated;
- what belongs in Mission Core and how admission is decided;
- which semantics are canonical and where they are owned;
- how domain missions may extend the framework without silently redefining Core.

## Semantic ownership

`mission-framework` owns canonical semantics and normative invariants.

`Mission-Platform` implements those semantics through schemas, contracts, validators, tools and interfaces.

Domain mission repositories apply, extend and challenge the semantics through real missions and evidence.

No implementation or domain repository may silently redefine a canonical concept.

## v0.2 workstreams

### 1. Semantic consolidation

- Maintain one canonical glossary/dictionary.
- Maintain one canonical Mission Loop.
- Treat alternative diagrams as explicitly named views, not competing canonical loops.
- Remove or redirect duplicate definitions.
- Separate normative statements from descriptive observations.

### 2. Mission Ontology and Dictionary

For each candidate canonical concept, document where applicable:

- definition;
- purpose;
- identity;
- lifecycle;
- attributes;
- relationships;
- invariants;
- examples;
- anti-patterns;
- extension rules.

Candidate concepts must not become Core merely because they are useful. Admission follows the Mission Core admission procedure.

Concepts requiring explicit investigation include, but are not limited to:

- Claim;
- Delegated Authority;
- Constraint;
- Uncertainty;
- Communication;
- Failure;
- Value;
- Commitment.

These are hypotheses, not predetermined Core members.

### 3. Comprehension and conformance

Test v0.2 with independent readers and implementations.

Reviewers should be able to identify the same canonical source and produce materially compatible interpretations without private context.

Machine-readable representations and platform implementations should be testable against semantic invariants rather than prose resemblance.

### 4. Reality validation

Mission Framework must be changed when credible evidence from real missions exposes a material flaw, omission or contradiction.

The framework must not force reality to fit its model.

A domain mission that cannot be represented cleanly should record a **Framework Finding** rather than hide the mismatch through local workarounds.

## Release criteria

v0.2 may be released when:

1. canonical semantic ownership is explicit;
2. the canonical Mission Loop is singular and identifiable;
3. canonical concepts have stable definitions and key relationships;
4. Core admission is procedural and evidence-based;
5. Evidence, Claim, Decision and Authority semantics are sufficiently precise for traceable implementation;
6. independent reviewers can reach materially compatible interpretations without conversation history;
7. known contradictions are either resolved or explicitly recorded as open issues;
8. the framework is ready to be challenged by a real mission without requiring hidden context.

## What v0.2 does not claim

v0.2 does not claim:

- universal completeness;
- final ontology;
- proof across all domains;
- that every useful concept belongs in Core;
- that Mission Platform is feature complete;
- that Mission Solar Eclipse has validated the framework before the mission produces evidence.

v0.2 is a semantic baseline designed to be challenged.

## The reality discipline

A strong framework does not grow primarily by accumulating documents.

It grows when real missions expose what the model cannot yet express, when those findings are supported by evidence, and when the framework changes accordingly.

> **Do not make the mission fit the framework. Make the framework answer to reality.**

This discipline is closely aligned with the expression **“Remtræk til Virkeligheden”**, attributed in this project to **Morten Thrane**.
